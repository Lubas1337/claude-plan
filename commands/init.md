# /claude-plan:init — Создание нового плана

## Инструкции

> При вопросах пользователю — используй `AskUserQuestion` tool с опциями (селектор), а не текстовые вопросы.

Ты создаёшь новый план в `.plan/`. Skill "plan" содержит форматы файлов и конвенции. Используй его как справку.

### Step 0: Detect Backend

Выполни процедуру **"Detect Backend"** из SKILL.md:
1. Проверь `.plan/config.json` → определи `mode` (local/obsidian) и `vaultPath`
2. Если `mode === "local"` → текущее поведение без изменений, все шаги ниже как есть
3. Если `mode === "obsidian"` → используй Obsidian-специфичные пути и форматы (см. ветвления ниже)

### Шаг 0: Выбор режима

Спроси пользователя через `AskUserQuestion` с опциями:
- **"Обсудить детально"** — research кодебазы + итеративное обсуждение (3 раунда) перед созданием плана
- **"AI решает всё"** — AI исследует молча, предлагает готовый план, пользователь подтверждает 1 раз
- **"Быстрый план"** — без исследования, сразу спросить goals/parts (классическое поведение)

Сохрани выбранный режим — он определяет, какие шаги выполняются.

### Шаг 1a: Загрузи проектный контекст

Выполни процедуру **"Project Context"** из SKILL.md:
- Из CLAUDE.md (уже в контексте): извлеки Stack, Project Structure, Architecture, Anti-Patterns
- Прочитай project memory (`~/.claude/projects/<path-hash>/memory/MEMORY.md`) — прошлые решения
- `ls .claude/skills/` — доступные навыки для аннотаций задач
- `ls internal/` (или исходную директорию) — реальные пакеты

Используй собранный контекст при разбиении на части, заполнении Files Affected и создании задач.

### Шаг 1b: Спроси у пользователя (упрощённый ввод)

Спроси у пользователя (если не указано в аргументах):
1. **Название плана** (slug для директории, например `ai-features`, `lms-integration`)
2. **Описание** — что хочет сделать, свободная форма (1-3 предложения)

> Goals и parts НЕ спрашиваются здесь — они определяются после исследования (Step 1c/1d).
> Исключение: режим "Быстрый план" — goals/parts спрашиваются в Step 1d.

### Шаг 1c: Research & Discussion

> **Пропускается** в режиме "Быстрый план" → переходи сразу к Шагу 1d.

#### Общая часть (для "Обсудить" и "AI решает")

Выполни процедуру **"Pre-Plan Research"** из SKILL.md:
- Поиск в кодебазе по ключевым словам из описания (grep по internal/, pkg/, cmd/)
- Проверка go.mod / package.json — текущие зависимости
- Проверка `.plan/*/research/` и `adr/` — прошлые исследования и решения
- Проверка project memory — прошлые решения и контекст
- Если описание затрагивает 3+ домена → запусти Explore agent (sonnet) для глубокого исследования
- Если описание упоминает внешние API/библиотеки → web search для актуальной информации

Сохрани находки — они используются дальше и записываются в `000-pre-plan.md`.

#### Режим "AI решает всё"

1. Молча исследуй (процедура выше)
2. Автоматически определи vision, goals, parts на основе находок
3. Покажи пользователю предложенный план (vision + goals + таблица частей)
4. Спроси через `AskUserQuestion`:
   - **"План хороший, создавай"** → переходи к Шагу 1d с предзаполненными данными
   - **"Хочу обсудить"** → переключись в режим "Обсудить детально", начни с Раунда 2
   - **"Хочу изменить"** (Other) → прими правки, покажи обновлённый план, спроси снова

#### Режим "Обсудить детально" — 3 раунда

**Раунд 1: Результаты исследования**

Покажи пользователю структурированные находки:
- Паттерны кодебазы, существующие пакеты и модули
- Зависимости (go.mod / package.json)
- Прошлые ADR и исследования
- Edge cases и потенциальные риски
- Релевантный контекст из project memory

Автоматически переходи к Раунду 2.

**Раунд 2: Обсуждение подхода**

AI предлагает:
- Архитектурное направление (с учётом находок)
- Грубую разбивку на части (без деталей)
- Ключевые trade-offs

Спроси через `AskUserQuestion`:
- **"Подход хороший"** → переходи к Раунду 3
- **"Покажи альтернативы"** → покажи альтернативный подход, спроси снова (макс 2 под-цикла, потом стир к решению)
- **"У меня другая идея"** (Other) → прими идею пользователя, адаптируй, переходи к Раунду 3

> Максимум 2 итерации альтернатив в Раунде 2. После 2-х предложи выбрать лучший вариант.

**Раунд 3: Структура плана**

AI предлагает конкретный план:
- Vision (1 предложение)
- Goals (2-5 целей)
- Таблица частей: название, зависимости, размеры, параллельность

Спроси через `AskUserQuestion`:
- **"План хороший, создавай"** → переходи к Шагу 1d
- **"Хочу изменить"** (Other) → прими правки, покажи обновлённый план, спроси снова
- **"Нужно больше research"** (Other) → выполни точечный мини-research по конкретному вопросу (не полный повтор Step 1c), вернись в Раунд 3

### Шаг 1d: Финализация

#### Режим "Быстрый план"

Спроси у пользователя (классическое поведение):
1. **Vision** — одно предложение, что план достигает
2. **Goals** — 2-5 целей
3. **Части** — список частей с названиями, зависимостями и размерами (S/M/L/XL)

Если пользователь даёт описание в свободной форме — сам разбей на части, учитывая архитектурные слои проекта.

#### Режим "AI решает" / "Обсудить"

Используй результаты Step 1c (vision, goals, parts уже определены).

Покажи финальную структуру плана и спроси через `AskUserQuestion`:
- **"Создать план"** → переходи к Шагу 2
- **"Хочу обсудить"** → переключись в режим обсуждения (Раунд 2)
- **"Изменить"** (Other) → прими правки, покажи обновлённый вариант, спроси снова

### Шаг 1a.1: Проверка PROJECT.md

Проверь `.plan/PROJECT.md`:
- Если **есть** → прочитай, извлеки Mission, Vision, Strategic Goals, Constraints, Roadmap
- Если **нет** → пропусти

Контекст PROJECT.md используется в шагах 1c и 1e.

### Шаг 1e: Alignment check (только при наличии PROJECT.md)

Если `.plan/PROJECT.md` существует:

1. **Стратегическая цель**: определи, какую цель из PROJECT.md этот план поддерживает.
   Покажи: "Этот план поддерживает цель: <goal>".
   Если ни одна не подходит — предупреди.
   Спроси через `AskUserQuestion`:
   - "Ок, создать" → продолжить
   - "Добавить цель в PROJECT.md" → предложи формулировку, добавь, продолжить

2. **Constraints**: проверь constraints из PROJECT.md. Если конфликт — предупреди.

3. **Roadmap**: предложи фазу roadmap для нового плана.
   Спроси через `AskUserQuestion`:
   - Опции — существующие фазы из Roadmap + "Новая фаза" + "Пропустить"
   Если выбрана фаза → добавь план в Roadmap таблицу PROJECT.md.

### Шаг 2: Создай структуру

```
.plan/
├── INDEX.md                          # Создай или обнови
└── <plan-name>/
    ├── MASTER.md
    ├── STATUS.md
    ├── CHANGELOG.md
    ├── research/
    │   ├── 000-pre-plan.md           # ← NEW: только если Step 1c выполнялся
    │   └── adr/
    └── parts/
        ├── 01-<slug>/
        │   └── META.md
        ├── 02-<slug>/
        │   └── META.md
        └── ...
```

> `000-pre-plan.md` создаётся только в режимах "Обсудить детально" и "AI решает всё". Формат — из SKILL.md.

#### Obsidian mode (Step 0: obsidian)

Если mode === "obsidian", вместо `.plan/<plan-name>/` создай в vault:

```
<vault_path>/Plans/<Plan Name>/
├── <Plan Name>.md                     # Plan Overview с YAML frontmatter
├── Status.md                          # Dataview дашборд
├── Changelog.md
├── Research/
│   ├── 000 - Pre-Plan.md             # Только если Step 1c выполнялся
│   └── ADR/
└── Parts/
    ├── 01 - <Name>/
    │   └── 01 - <Name>.md            # Part note с frontmatter
    ├── 02 - <Name>/
    │   └── 02 - <Name>.md
    └── ...
```

Plans MOC.md (`<vault_path>/Plans/Plans MOC.md`) — уже содержит Dataview, не требует обновления.

### Шаг 3: Заполни файлы

1. **MASTER.md** — по формату из SKILL.md. Таблица частей с зависимостями.
2. **META.md** для каждой части — статус `ready` если нет зависимостей, `blocked` если есть незавершённые.
3. **STATUS.md** — начальный дашборд (все blocked/ready).
4. **CHANGELOG.md** — пустой шаблон с `# Changelog`.
5. **INDEX.md** — добавь новый план в таблицу (создай файл если не существует).
6. **000-pre-plan.md** — если Step 1c выполнялся, запиши находки и обсуждение по формату из SKILL.md.

#### Obsidian mode files

В Obsidian mode используй форматы из SKILL.md секции "Obsidian File Formats":
1. **Plan Overview** (`<Plan Name>.md`) — YAML frontmatter с `type: plan`, wikilinks в Parts таблице
2. **Part Notes** (`NN - Name.md`) — frontmatter с `type: part`, `plan: "[[Plan Name]]"`, `status:`, `dependencies:[]`
3. **Status.md** — Dataview-based (формат из SKILL.md)
4. **Changelog.md** — тот же формат что local
5. **Plans MOC** — Dataview, уже создан при setup
6. **000 - Pre-Plan.md** — frontmatter с `type: research`, wikilink на план

### Шаг 4: Задачи для частей

Для каждой части создай task files в `parts/NN-slug/`:
- Разбей часть на 2-5 конкретных задач
- Каждая задача — отдельный файл `01-task.md`, `02-task.md`
- Формат task file — из SKILL.md

#### Agent Assignment

Для каждой создаваемой части предложи значение `Agent:`:
- **Size S** → `Agent: none` (дешевле выполнить inline)
- **Size M с Delegatable: yes** → `Agent: auto`
- **Size L/XL** → `Agent: auto` (рекомендуй делегирование для больших частей)
- Если часть research-focused → `Agent: Explore`
- Если часть содержит архитектурные решения → `Agent: architect`

Добавь колонку Agent в таблицу частей MASTER.md и в META.md каждой части.

Спроси пользователя через `AskUserQuestion` с опциями:
- "Назначить агентов" — назначить агентов для частей по рекомендациям выше
- "Пропустить" — не назначать агентов (можно добавить позже в META.md)

#### Obsidian mode tasks

В Obsidian mode задачи — отдельные notes с YAML frontmatter:
- Файл: `Parts/NN - Name/NN - Task Name.md`
- Формат: `type: task`, `plan: "[[Plan Name]]"`, `part: "[[NN - Name]]"`, `task_number: "NN"`
- В таблице Tasks в Part note используй wikilinks: `[[NN - Task Name]]`

### Правила

- Slug плана: lowercase, дефисы, без пробелов (`ai-features`, не `AI Features`)
- Номера частей: двузначные с нулём (`01`, `02`, ..., `10`)
- Первая часть без зависимостей — `ready`, остальные — `blocked` или `ready` по графу
- Не создавай task files если пользователь не описал задачи детально — оставь только META.md
- Добавляй `Agent:` в META.md только если пользователь подтвердил назначение агентов
- Размер (S/M/L/XL): S = 1-2 задачи, M = 3-5, L = 6-10, XL = 10+
- Files Affected в META.md — используй реальные пути из структуры проекта (не выдумывай)
- Задачи по конвенциям: если часть создаёт новый пакет — добавь задачу на `doc.go`
- При наличии релевантных skills — добавь `Skill hint: <name>` в Notes секции META.md
- Учитывай Anti-Patterns из CLAUDE.md при формулировке задач

### Вывод

После создания покажи:
1. Дерево созданных файлов
2. Таблицу частей из MASTER.md
3. Подсказку: `/claude-plan:next <plan-name>` для начала работы
